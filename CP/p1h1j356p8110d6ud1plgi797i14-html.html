1. 

Ведение 

Выполняя задания из данного курса вы научитесь создавать HA-кластер Positive 
Technologies Application Firewall с помощью визарда и через CLI WSC.  

Полученные навыки помогут вам распределить нагрузку между узлами кластера Positive 
Technologies Application Firewall.  

2. 

Инфраструктура учебного стенда 

2.1. 

Описание учебного стенда 

 

Каждому ученику предоставляется персональный Учебный стенд, расположенный в 
инфраструктуре облачного провайдера. Управление стендом осуществляется с сервера 
удалённых рабочих столов ( RDS ):  rds.edu.tsecurity.com.  

 

После подключения к RDS серверу учащемуся доступны три сервера Positive Technologies 
Application Firewall  и сервер с уязвимыми приложениями: vulnbank.local, news.local  и 
mu.local.  

2.2. 

Сетевые интерфейсы Application FireWall  

 


 

3

 

Каждый экземпляр Positive Technologies Application Firewall имеет четыре сетевых 
интерфейса:  
eth-mgmt  - предназначен для управления Application firewall через SSH и веб-
консоль. Данный интерфейс подключен к сети с DHCP-сервером с диапазоном 
адресов 10.0.1.0/24.  

eth-cluster - предназначен для соединения с базами данных при работе в кластере. 
На интерфейсе задается статичный IP-адрес из сети 192.168.1.0/24.   

eth-ext1 - предназначен для приема входящих подключений со стороны клиентов. 
Интерфейс подключен к сети с DHCP-сервером с диапазоном адресов 
10.80.160.0/24 

eth-int1 - предназначен для подключения к защищаемым серверам. На интерфейсе 
задается статичный IP-адрес из сети 192.168.2.0/24 

Доступ в интернет со стендов Application Firewall: 

Шлюз для выхода в интернет находится на интерфейсе eth-ext1 и имеет адрес 
10.80.160.1 

Доступ к PT AF через интернет есть только по портам 80, 443 и 8443. 

2.3. 

Учетные записи 

Доступ по SSH возможен только через интерфейс eth-mgmt по порту 22013. Для 
подключения можно использовать утилиту PuTTY, доступную из папки Utilities на 
виртуальном рабочем столе. Учетные данные для доступа таковы: 

Login: pt 

Password: positive 

Оболочкой по умолчанию является wsc: утилита, предназначенная для настройки 
сети. 

Доступ к веб-консоли возможен (до специфической настройки) с помощью любого 
браузера из любой сети по порту 8443 со следующими учетными данными: 

Login: admin 

Password: postive (по умолчанию, требует изменения после первого обращения). 

2.4. 

Доступные приложения 

Все приложения развернуты в локальном домене .local и доступны напрямую 
только изнутри виртуального учебного класса. 


 

4

 

Приложения не могут быть открыты для прямого доступа через интернет. 

Доступны три защищаемых приложения: 

mu.local: приложение OWASP Mutillidae 2; 

vulnbank.local: сайт «банка» с уязвимым приложением онлайн-банкинга; 

news.local: новостной сайт, уязвимый к SQL-инъекциям и XSS; 

Сайты vulnbank.local и mu.local поддерживают авторизацию. Если в описании 
практической работы не оговорено иное, то для авторизации используйте 
пользователя с именем user** и паролем P@ssw0rd, где ** - это номер 
выделенного вам стенда, от 01 до 15 (если у вас номер из одной цифры, добавьте 
слева ноль). 

Также доступен «зловредный» сайт evilsite.local, на котором собраны ресурсы, 
используемые в атаках на защищаемые сайты и приложения. 

Для обращения к защищаемым приложениям снаружи через Application Firewall 
необходимо внести изменения либо в файл hosts вашего физического компьютера, 
на котором вы работаете, либо в файл зоны .local доступного ей DNS-сервера. 

При внесении изменений надо поставить в соответствие именам приложений 
внешний IP-адрес экземпляра Application Firewall. 

Пример дополнительной строки для файла hosts (следует указать корректный IP-
адрес): 

<внешний адрес AF> mu.local vulnbank.local news.local 
evilsite.local 

Доступ к evilsite.local требуется только потому, что приложение, открываемое через 
Application Firewall в клиентском браузере, должно иметь возможность к нему 
обращаться.

3.2. 

Настройка  виртуальных  маршрутизаторов  и  механизмов  обработки 

трафика. 

 

1.  Откройте веб-консоль Application Firewall, пройдя по URL в приглашении в консоли узла 

master. Авторизуйтесь, используя логин admin и пароль positive.  

 

2.  На первой странице мастера смените пароль: введите один раз действующий пароль и 

два раза новый. 

 

3.  На следующей странице в промышленных условиях нужно скачать fingerprint’ы машин и 

загрузить файлы лицензий, но в лабораторных условиях лицензия раздается по сети и в 
этих операциях нет необходимости.  


 

15

 

• Есть нюанс. Если вы переименовывали виртуальные машины в процессе конфигурации 
более одного раза, то может возникнуть ситуация, когда у вашему PTAF выдадут более 
одной лицензии.  

 

Это потому что изменение hostname не удаляются из коллекции Gateways в Mongo DB. И 
теперь лишнюю запись нужно удалить из базы вручную.

Настройка кластера в ручном режиме

4.1. 

Добавление третьего узла к настроенному кластеру из двух узлов  

1. Подключитесь к третьему экземпляру Application Firewall и войдите в систему. В ответ на 
вопрос о смене hostname укажите имя ptaf-**-t, где ** - номер вашего рабочего места (если 
цифра одна, добавьте слева ноль).  

Далее в тексте ссылки на этот экземпляр будут как на tertiary.  

2. Дождитесь на tertiary окончания предварительной конфигурации, выйдите из мастера 
нажав 0, и проверьте настройку сетевого окружения, последовательно выполнив следующие 
команды:  


 

23

 

• Интерфейс eth-mgmt должен быть настроен в сети 10.0.1.0/24 через dhcp. Проверьте это 
выполнив команду:  

If list eth-mgmt 
• Интерфейс eth-ext1 должен быть настроен в сети 10.80.160.0/24 через dhcp. Проверьте это 
выполнив команду:  

If mode eth-ext1 dhcp  
 

• Для интерфейса eth-cluster настраивается статичный IP-адрес и его необходимо 
переопределить. Укажите адрес в сети 192.168.1.0/24, а в качестве последнего октета 
укажите (узел tertiary) номер вашего рабочего места плюс 100  

 

If set eth-cluster inet_address 192.168.1.1**  
ПРИМЕР: для рабочего места номер 5 адрес на интерфейсе eth-cluster узла должен быть 
задан как 192.168.1.105/24

укажите (узел tertiary) номер вашего рабочего места плюс 100  

 

If set eth-int1 inet_address 192.168.2.1**  
ПРИМЕР: для рабочего места номер 5 адрес на интерфейсе eth-int1 узла должен быть задан

на узле slave скрипт пройдет быстро и закончится, на узле tertiary он остановится с уже

Если скрипт на master не завершается, а ожидает config sync дочерних нод, просто по 
очереди выполните вручную config sync на slave и tertiary.
